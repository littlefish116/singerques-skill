# submissions/ · 新信息提交接口

本目录是知识库的**输入接口**：用户在使用中获得的**新的真实信息**，先到这里登记，经评审后再并入 `references/`。
设计目标：让知识库能**动态、正确**地更新——新信息先 *暂存待证 → 脱敏 → 守门 → 留痕*，再决定是否并入，**不直接覆盖定论**。

> 注意区分两类"用户信息"：
> - **你自己的真实情况**（听歌偏好/现场/动机等）→ 走 `SKILL.md` 第 0 步的"采集"流程，按会话使用，**不必**提交到这里。
> - **新的通用知识**（新题目/新雷区/规则变化/经验/勘误）→ 提交到本目录，供知识库更新。

---

## 怎么提交

**方式 0（公开仓库推荐）· GitHub Issue / PR**

本 skill 开源后，公开投稿走 GitHub：开一个 Issue（脱敏后的内容 + 类别 + 来源类型）→ 维护者复核 → PR 并入 `references/` + `CHANGELOG.md`。完整流程与守门规则见仓库根目录 `CONTRIBUTING.md`。下方方式 A/B 仍是本地 / 离线贡献的入口。

**方式 A（本地：结构化 + 自动脱敏）**

```bash
python3 scripts/submit.py \
  --category 电话面试常见问题 \
  --content "对方问我……" \
  --source 待证 \
  --by 匿名
```
脚本会自动：脱敏（电话/邮箱/wxid/身份证/链接/@昵称）→ 生成结构化条目 → 追加到 `_inbox.md`。

**方式 B（手动）**：复制 `_template.md`，填好后追加到 `_inbox.md` 末尾（手动提交请自行先脱敏）。

## 可提交的类别（`--category`）

8 个主题类：`问卷常见问题` / `电话面试常见问题` / `节目组筛选标准` / `回答中的高风险表述` / `成功入选者经验摘要` / `落选或被质疑原因摘要` / `现场听审注意事项` / `如何表达理性普通观众`

3 个元类别：`规则与流程变化` / `勘误与纠正` / `其他`

## 来源强度（提交时标，合并时复核）

| 标签 | 含义 |
|------|------|
| `高` | 多人群、跨期反复印证（**新提交默认不当高**，需评审交叉印证后才能升到高） |
| `中` | 个别一手陈述、内部自洽 |
| `低` | 孤立、单一来源、未佐证 |
| `推测` | 群友猜测、未经证实 |
| `待证` | **新提交的默认状态**，等复核 |

## 守门规则（以下会被拒收，不进知识库）

- ❌ 任何"如何**编造经历 / 规避审查 / 伪装身份 / 隐藏社交账号**"的内容——违反 `SKILL.md` 第 5 节禁止事项。
- ❌ 含**未脱敏隐私**的：电话/邮箱/wxid/身份证/真实姓名/社交链接/群昵称（提交前必须脱敏）。
- ❌ 与报名/面试表达无关的**纯八卦、赛果讨论、情绪宣泄**。

## 评审与合并流程（"正确更新"）

1. `_inbox.md` 中每条状态：`待审` → 复核后改为 `已采纳` 或 `已驳回`（驳回请简注原因）。
2. **采纳**的条目：交叉印证后确定来源强度，并入对应 `references/` 文件，并在 `../CHANGELOG.md` 记一笔（日期 + 类别 + 摘要 + 强度）。
3. **不自动覆盖**已有的 `高` 可信结论；`勘误与纠正` 类单独标注原条目。
4. 合并时一律标注 `高`/`中`/`低`/`推测`，与现有 references 口径一致。

## 文件

- `_template.md` —— 提交模板（手动提交用）
- `_inbox.md` —— 待审区（所有提交都追加到这里）
